# 大型任务执行协议

## 建立任务图

先识别目标、不变量、依赖、风险和验收，再拆任务。按依赖图排序，不按文件列表机械拆分。

```text
确认目标与不变量
→ 定位受影响接口
→ 建立失败复现或验收检查
→ 实施最小改动
→ 运行局部验证
→ 检查跨模块影响
→ 运行完整验证
→ 更新状态和交付
```

只有输入充分、验收独立、修改范围不重叠的任务才适合并行。共享核心接口或相同文件的任务应串行或先固定接口。

## 原子任务契约

```text
目标：
前置依赖：
输入上下文：
允许修改范围：
必须保持的不变量：
预期产物：
验证命令或证据：
成功标准：
失败分类：
恢复点：
```

无法定义独立验收的任务继续拆分。不要把“完成整个功能”交给一个没有中间检查点的执行循环。

## 执行循环

按以下循环工作：

1. 读取最小上下文；
2. 提出一个可证伪假设；
3. 预测改动后的可观察结果；
4. 实施最小改动；
5. 运行最小相关验证；
6. 比较预测与实际；
7. 分类失败；
8. 继续、回退或升级。

一次循环只检验一个主要假设。不得用长篇推测代替可以运行的检查。

## 错误处理

先分类，再修复：问题理解、假设、实现、测试/数据、环境/依赖、上下文或权限。若错误类型改变，重新建立假设；不要延续上一轮修复惯性。

设置循环限制：

- 同一假设连续失败时重新诊断；
- 扩大修改范围前重新确认任务边界；
- 无法建立验证手段时停止实现；
- 遇到用户的重叠修改、权限扩大或破坏性操作时按授权规则处理；
- 达到用户设定的时间、成本或尝试上限时升级。

不要硬编码通用次数阈值；根据失败成本和任务预算设定。

## 检查点与恢复

每个原子任务完成后记录：改了什么、为什么、验证结果、遗留问题和下一步。优先使用可恢复的补丁、版本提交或明确状态文件。恢复时先核对当前文件和外部状态，不假定中断前状态仍成立。

## Token控制

- 先搜索，再读取相关片段；
- 优先读取定义、调用方、测试和当前错误；
- 已确认事实只保留压缩摘要；
- 不重复解释已完成步骤；
- 长资料按需引用，不整份加载；
- 子任务只携带自己的上下文包；
- 先运行低成本局部验证，再运行完整验证；
- 错误日志保留根因相关片段和命令环境。

Token减少不能以遗漏约束、降低验证或增加返工为代价。
