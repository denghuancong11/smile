# 上下文与工具路由

## 目标

获取完成当前任务所需的最小充分上下文，而不是最大化读取量。插件和工具只有在弥补已确认的上下文缺口时才接入。

仓库任务在阶段1运行只读的 `project_context.py build`，生成文件类型、配置、文档、测试和样本文件清单。扫描被截断或没有识别到项目配置时回执失败，必须缩小根目录、提高上限或人工说明项目类型后重跑。纯概念任务使用 `skip` 并写明原因。

在推荐具体插件前，先确认：主要任务、上下文实际存放位置、当前工具、权限边界和已观察缺口。缺少这些信息时，先问一个最能决定上下文路由的问题并停止；不得给出具体插件名称、品牌矩阵或“最佳组合”。

## 上下文地图

| 类型 | 典型来源 | 主要风险 |
|---|---|---|
| 代码结构 | 文本搜索、符号索引、语言服务器 | 只看定义而遗漏调用方 |
| 修改历史 | Git、提交、PR | 历史结论已经失效 |
| 设计决策 | README、ADR、知识库 | 文档过时或无所有者 |
| 需求状态 | Issue、项目管理平台 | 需求与代码状态不同步 |
| 运行状态 | 终端、测试、日志、监控 | 日志噪声和环境差异 |
| 外部规范 | 官方文档、API、依赖源码 | 版本不匹配 |

## 评估来源

对每个来源记录：权威性、新鲜度、检索粒度、Token成本、可追溯性、权限、故障方式和替代来源。不要把搜索结果摘要当作原始证据。

## 默认路由

```text
找代码位置 → 先文本或符号搜索
理解行为 → 读取定义、调用方和测试
确认变更原因 → 查询Git和设计记录
定位当前错误 → 读取最新命令、日志和环境
确认依赖用法 → 匹配版本的官方文档或本地源码
确认外部任务状态 → 对应连接器或项目系统
```

优先使用项目内权威来源；存在时间敏感性时验证新鲜度。

## 最小充分上下文包

为每个原子任务提供：

```text
目标与验收：
相关文件或对象：
关键接口与不变量：
允许修改范围：
近期相关变更：
已有设计决策：
当前错误或验证结果：
来源与时间：
```

完成后保留结论、证据和变更摘要，丢弃无关原始材料。

清单只用于定位下一批权威文件，不能替代读取定义、调用方、测试、配置和当前错误证据。

## 插件选择流程

1. 用基线工具完成代表性任务；
2. 定位缺失的上下文能力；
3. 每次只增加一个插件、连接器或索引层；
4. 在同一任务集上比较遗漏、噪声、Token、延迟和维护成本；
5. 保留有稳定增益的工具；
6. 记录权限、数据范围、过期机制和故障降级路径。

询问具体插件推荐时，检查当前环境和最新官方能力。不存在跨项目通用的最佳工具栈。

即使用户要求“直接给最佳组合”，也先取得上述事实。只有用户明确要求不继续诊断时，才提供不含品牌的能力类别基线，并列出尚未确认的环境事实。

## 本地项目的最小基线

在增加复杂基础设施前，确认是否已经具备：

- 快速文本搜索；
- Git历史与差异查看；
- 构建、测试、格式化和静态检查命令；
- 语言或符号导航；
- 项目说明、架构决策和环境配置；
- 可定位到具体任务的错误输出。

知识库适合保存跨会话决策与背景，不应替代代码、测试和版本历史这些一手事实。

## 失败与降级

连接器不可用、索引过期或权限不足时：标记缺口，切换到权威替代来源，缩小结论范围。不得用猜测补全缺失上下文。
