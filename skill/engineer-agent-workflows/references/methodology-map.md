# 原方法论与十阶段映射

## 不可丢失的判断原则

1. Idea的价值在于带来新视角、机制问题或被忽视的重要问题，而不只是指标提升。
2. 从“更快的马”式表面表达连续追问到底层目的、失败机制和真正能力。
3. 没有明显问题时先复现强基线，再用真实风险相关的扰动主动暴露空白。
4. AI适合检索、总结、枚举、实现和试验；人必须判断方向、价值、风险和取舍。
5. 强基线上的工程优化不自动等于创新；必须说明修复了什么机制及其边界。
6. 好Story包含真实且重要的问题、反直觉切入点、针对机制的MVP和问题—方法—证据闭环。
7. 专业术语、热点和“插件到范式”只能表达真实贡献，不得制造或夸大贡献。
8. 已验证机制可以纵向深化、横向迁移和研究边界，但每个扩展都需独立证据。
9. 资源有限时优先做重要、可验证、资源匹配而强团队投入较少的问题。

## 十阶段对应

| 阶段 | 原方法论作用 | 防偏航要求 |
|---|---|---|
| 对齐 | 从表面表达回到底层目的 | 先固定约束和验收，不急于给解法 |
| 回忆 | 利用已有失败和先发积累 | 实际检索，不把“我记得”当证据 |
| 探索 | 本质追问、扰动、反直觉切入 | 至少三种算子与三个候选问题 |
| 质疑 | 判断Idea而非迎合Idea | 主动找反例、冲突和否定证据 |
| 决策 | 人判断什么能做、不能做 | 人明确选择目标、方案、风险边界 |
| 验证 | 不完美但针对机制的MVP | 冻结标准后执行最小验证 |
| 复盘 | 失败也要记录并解释 | 区分对齐、假设、执行、数据错误 |
| 沉淀 | 形成自己的研究圈和经验资产 | 写入可检索记录与适用边界 |
| 回归 | 用证据维护逻辑闭环 | 把关键错误变成机器检查 |
| 循环 | 利用先发优势持续迭代 | 下一任务实际检索旧记录后再行动 |

## 三类Agent问题的转换

### 提示和Skill

不要从万能提示开始。先定位目标、状态、上下文、工具和验证机制的首次失效；探索多种机制并找反证；人选择后才把验证成功的做法固化到系统指令、仓库规则或Skill。

### 插件和生态

不要从插件名单开始。先检索历史工具决策，画上下文地图，探索缺口的不同解释，再比较命中、噪声、Token、延迟、权限和维护成本。只保留在固定任务集上有证据增益的工具。

### 大型工程任务

不要按文件机械拆分。围绕目标、不变量、依赖和独立验收建立任务图；每轮只检验一个可证伪假设；复盘错误类别；把重复错误转成测试、静态检查、状态断言或恢复测试。

## 结论边界

- 未检索最新资料时，不宣称首次、市场空白、最佳插件或当前SOTA。
- 未完成最小验证时，只能称候选或待验证方案。
- 单次成功不能推出通用可靠；记录任务、环境、模型、工具和扰动边界。
- 历史记录是证据候选，不是当前结论；必须在阶段2判断适用性。
