# Skill与提示协议

## 适用边界

用于设计或评审系统提示、仓库级指令、AGENTS.md、垂类Skill和长任务状态协议。先确认问题确实属于指令或状态表达；若根因是上下文缺失、工具不可用或任务不可验证，转到对应模块。

面对“给我一份万能提示”时，先明确不存在跨任务保证有效的万能提示，再把症状转成行为目标。缺少失败轨迹时，只提供行为契约字段、最多一项最小干预和采集方法，不输出可直接复制的完整系统提示。取得至少一个可复现失效后再定制。任何提示在固定任务集和扰动测试通过前都标记为基线草案。

## 选择承载层

| 信息类型 | 推荐位置 |
|---|---|
| 长期不变的角色、安全和授权边界 | 系统级指令 |
| 仓库约定、构建与验证方式 | AGENTS.md或仓库指令 |
| 特定领域的可复用工作流 | Skill |
| 当前任务目标、进度和失败记录 | 外部状态文件或任务状态 |
| 大篇幅领域资料和模板 | Skill的references或assets |

不要把易变项目事实写进系统提示，也不要把长期安全规则只保存在临时对话中。

## 行为契约

为Agent定义可检查契约：

```text
目标结果：
输入及其来源：
必须遵守的约束：
允许执行的动作：
必须取得确认的动作：
禁止行为：
阶段与完成条件：
验证要求：
失败和升级条件：
输出格式：
```

避免“认真思考”“不要忘记”“尽量正确”等无法验收的措辞。将它们改写为行为，例如“修改后运行指定测试并报告结果；测试未运行时不得宣称完成”。

## 状态机

对超过一个执行阶段的任务，维护：

```text
当前目标：
当前阶段：
已确认事实：
暂定假设：
已完成事项：
修改过的对象：
验证结果：
未决问题：
下一项最小行动：
停止或升级条件：
```

只在状态变化时更新。中断、压缩或换执行者后，先读取状态，再继续工作。状态记录保存结论和证据，不保存冗长思维过程。

## 指令结构

按以下顺序编写：

1. 结果与适用范围；
2. 授权和硬约束；
3. 输入检查与缺失信息处理；
4. 工作流和阶段完成条件；
5. 工具与文件操作规则；
6. 验证、错误处理和恢复；
7. 状态维护；
8. 最终输出契约。

删除重复强调、人格修饰和不影响行为的背景。若两条规则冲突，明确优先级或缩小适用条件。

## Skill设计

SKILL.md只保留路由、核心流程和必须遵守的规则。将不同场景的详细协议拆到一层references中，并明确何时读取。只有重复且需要确定性的计算或文件处理才添加scripts。

Skill的description同时写清：做什么、何时触发、典型用户表达。正文不要依赖触发前才能知道的信息。

## 评审问题

- 目标能否被外部观察？
- 每个阶段是否有完成条件？
- Agent如何知道自己跑偏？
- 上下文缺失、冲突或过期时怎么办？
- 工具失败和测试失败如何分类？
- 哪些操作必须由用户确认？
- 中断后从哪里恢复？
- “完成”是否有证据？
- 规则能否通过测试，而不是只能主观评价？

## 最小测试

至少测试：正常任务、缺失输入、冲突指令、长上下文、中断恢复、工具失败和未经验证的完成声明。每次改动后重跑固定任务集，防止提示修复一个问题又引入另一个问题。
