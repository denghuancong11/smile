# 评估与回归

## 评估设计

建立基线、干预和固定任务集。任务集必须包含常规任务、已知失败任务、未用于设计的保留任务和中断/噪声/权限等扰动任务。一次主要实验只改变一个机制。

用户定义重要性和阈值；不要自行发明通用及格线。

阶段9使用 `evaluation_harness.py` 比较同一任务集上的基线与候选。工具强制检查成功率、约束违反、恢复率、逐任务退化和Token增幅阈值；任一阈值失败就生成失败回执，守卫拒绝推进。

## 指标

| 指标 | 定义 |
|---|---|
| 任务成功 | 是否满足预先定义的验收条件 |
| 约束违反 | 越权、范围扩大、遗漏硬约束的次数 |
| 首次通过 | 无需返工即通过验证的比例 |
| 修复轮次 | 从首次失败到通过所需循环数 |
| 重复工作 | 重复读取、分析或修改已完成内容 |
| 上下文命中 | 所需关键上下文是否被及时检索 |
| 上下文噪声 | 加载但未用于任务的信息量 |
| 恢复成功 | 中断后是否正确恢复目标和进度 |
| Token成本 | 完成任务的输入输出Token及其分布 |
| 时间和工具成本 | 完成任务的时长、调用和外部费用 |

Token应与成功、返工和错误成本一起评价。低Token但成功率下降不是优化。

## 证据闭环

| 失效或主张 | 根因假设 | 干预 | 证据 | 结论边界 | 状态 |
|---|---|---|---|---|---|
| | | | | | 已证实/部分支持/未验证/被否定 |

只在已测试的任务、工具和扰动范围内下结论。不能从少量样本推出“永不跑偏”或“通用最佳”。

## 回归流程

1. 保存能复现旧问题的任务；
2. 加入正常任务防止过拟合；
3. 加入保留任务检验泛化；
4. 记录模型、工具、版本、提示、Skill和环境；
5. 每次协议变更后重跑；
6. 比较成功、约束、恢复、Token和时间；
7. 若关键指标退化，回滚或重新诊断。

## 最终决策

输出保留、修改或放弃，并说明证据。将验证成功的机制固化到最小适当层级，不把整个实验过程永久塞进系统提示。
